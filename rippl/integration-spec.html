Pipedrive → Klaviyo

Integration spec

Four deliverables. Identity properties on the Klaviyo profile. Three custom events that carry the deal payload. Five rolling lifecycle properties the middleware writes on every event. And a one-time backfill across existing Pipedrive deals. Once this is in place, any prospect, customer, open-deal contact, or lapsed account is a one or two-condition segment in Klaviyo.

What we need built

01
Identity properties on the Klaviyo profile
Profile · stable
Eight person-level fields synced from Pipedrive to the Klaviyo profile. Stable, safe to overwrite. These describe who the person is, not what they ordered.
02
Three custom events carrying the deal payload
Events
B2B Inquiry Submitted fires on every Pipedrive Deal · added (= every form submission). B2B Order Placed fires when a deal moves to Won. B2B Deal Lost fires when a deal moves to Lost. All three carry the deal payload inline. All three dedup on pipedrive_deal_id.
03
Five rolling lifecycle properties on the Klaviyo profile
Profile · computed
Summary properties (b2b_lifecycle_stage, b2b_open_deal_count, b2b_total_orders, b2b_last_order_date, b2b_clv) written by the middleware on every event. With lifecycle held on the profile, Klaviyo segments don't need to join events at query time.
04
One-time backfill across existing Pipedrive deals
Backfill
Two passes. Pass 1: emit one event per historical deal (Inquiry for every deal, Order for every Won, Lost for every Lost). Pass 2: replay the event stream per contact and write the final rolling-property state to the profile. Same dedup key, so re-runs are safe.

Deliverable 03 · Rolling lifecycle properties

Five summary properties on the Klaviyo profile. The middleware writes them on every event. Because the lifecycle state lives on the profile, Klaviyo segments read it directly without joining events at query time.

PropertyTypeNotes
b2b_lifecycle_stageenumLead / Active Inquiry / Customer, recomputed every event (rule below)
b2b_open_deal_countinteger+1 on Inquiry, −1 on Order or Lost. Never below zero
b2b_total_ordersinteger+1 on Order. Lifetime, not time-bounded
b2b_last_order_dateISO dateSet to order_date on every Order. Used for win-back / lapsed segments
b2b_clvnumber (AUD)Cumulative sum of deal_value across all Orders
State machine · counter changes on each event
On eventCounter changesLifecycle stage
B2B Inquiry Submittedopen_deal_count + 1recompute (see below)
B2B Order Placedopen_deal_count − 1
total_orders + 1
last_order_date = order_date
clv += deal_value		Customer
B2B Deal Lostopen_deal_count − 1recompute (see below)
Lifecycle recompute rule:
· if total_orders > 0 → Customer
· else if open_deal_count > 0 → Active Inquiry
· else → Lead

A "Customer with an open new inquiry" stays Customer. Their open deal is captured in b2b_open_deal_count > 0 and segmentable separately. "Churned" is not a lifecycle stage; it's a segment query: b2b_lifecycle_stage = Customer AND b2b_last_order_date more than 365d ago.
How the side-effect update reaches Klaviyo: the rolling properties travel with the event itself. Each event POST to /api/events includes the recomputed profile properties in the profile.attributes.properties block, so one API call covers both the event write and the property update. To compute the next state, the middleware first GETs the current profile (/api/profiles/?filter=equals(email,"...")), reads the current counter values, applies the delta, and posts the event with the new values inline.

Deliverable 04 · Backfill

Run both event handlers and the property updater over existing Pipedrive data once, so the Klaviyo account is enriched on day one. Two passes: events first, then rolling properties replayed from the event stream.

  • Pass 1aOne B2B Inquiry Submitted event per existing Pipedrive deal, regardless of stage. Use the deal's creation date as submitted_at.
  • Pass 1bOne B2B Order Placed event per existing Won deal. Use the deal's Won date as order_date.
  • Pass 1cOne B2B Deal Lost event per existing Lost deal. Use the deal's Lost date as lost_at.
  • Pass 1dIdentity properties synced for every Pipedrive Person attached to a deal in the backfill set. b2b_lead_source set once, not updated.
  • Pass 2For each contact, replay the event stream chronologically and write the final state of all five rolling lifecycle properties to the profile.
  • DedupAll events keyed by pipedrive_deal_id, so re-runs are idempotent. Re-running the backfill never creates duplicates.

Why this model holds when a contact has multiple deals

The model is built so a contact can have any combination of open, won, and lost deals without anything being overwritten, and every lifecycle state sits behind a single segment query.

Klaviyo profile properties hold a single value per profile. If we synced per-deal data into them, every new inquiry or order would overwrite the last one, and Outfunnel's last-updated-wins behaviour would turn two simultaneous open deals into a race condition. The split model avoids both.

  • IdentityOnly person-level fields live on the profile (name, email, phone, company, job title, industry, lead source). These don't change deal-to-deal, so there's nothing to overwrite incorrectly.
  • Deal dataEverything specific to a deal (vessel, quantity, brief, dates, value) lives in the event payload. Each inquiry, each order, each Lost deal is a separate event keyed by pipedrive_deal_id.
  • LifecycleThe five rolling properties pre-compute lifecycle state. A contact with two open inquiries has b2b_open_deal_count = 2; a Customer with a new open deal has both b2b_lifecycle_stage = Customer and b2b_open_deal_count > 0.
  • Segments"Prospects" = b2b_lifecycle_stage = Lead. "Active clients" = = Customer. "Open deals right now" = b2b_open_deal_count > 0. "Lapsed customers" = = Customer AND b2b_last_order_date more than 365d ago. All one or two conditions.

01
Configure four Pipedrive webhooks
Pipedrive · webhooks
Point each webhook at a dedicated middleware endpoint (one per scenario / workflow).
  • Deal · added → Inquiry handler
  • Deal · updated, filter status = won → Order handler
  • Deal · updated, filter status = lost → Deal Lost handler
  • Person · added and Person · updated → Identity sync handler
02
Build the Inquiry handler scenario
Middleware · scenario 1
Receives the Deal · added payload from Pipedrive. Steps:
  • Look up the linked Person's email + the Org ID from Pipedrive (one or two API calls).
  • GET the Klaviyo profile to read current b2b_open_deal_count and b2b_total_orders.
  • Compute new open_deal_count (+1) and recompute b2b_lifecycle_stage.
  • POST B2B Inquiry Submitted to /api/events with the full event payload and the updated rolling properties inline.
03
Build the Order handler scenario
Middleware · scenario 2
Receives Deal · updated where status flipped to Won. Same shape as the Inquiry handler, but the counter changes are: open_deal_count − 1, total_orders + 1, last_order_date = order_date, clv += deal_value, lifecycle_stage = Customer. Also compute is_reorder by checking whether total_orders was already > 0 before this event.
04
Build the Deal Lost handler scenario
Middleware · scenario 3
Receives Deal · updated where status flipped to Lost. Counter changes: open_deal_count − 1; recompute lifecycle. Same POST pattern as the other two.
05
Build the Identity sync handler
Middleware · scenario 4
Receives Person · added or Person · updated. PATCH the Klaviyo profile with the eight identity properties. Skip b2b_lead_source if it's already set. That field is first-touch only.
06

Credentials
Tethrd generates and shares a private Klaviyo API key with events:write + profiles:write scopes. Store it as a Make.com Connection or n8n Header Auth credential. Never paste into Pipedrive, scenario JSON, or any plaintext location. Full handling guidance in the section below.
07
Run the backfill (two passes, idempotent)
Backfill
Pull all historical Pipedrive deals. Pass 1: fire one Inquiry per deal, one Order per Won, one Lost per Lost. Pass 2: replay the event stream per contact and write the final rolling-property state. 08
QA before go-live
Validation
Three checks before we cut over:
  • Dedup test: trigger a Won deal twice, confirm only one B2B Order Placed event lands.
  • Multi-deal test: on a test contact, create one Open deal, one Won, one Lost. Verify b2b_open_deal_count = 1, b2b_total_orders = 1, b2b_lifecycle_stage = Customer.

Handling the Klaviyo private API key

We will generate a Klaviyo private API key scoped to events:write and profiles:write only, and share it with Moti directly. The key must never be exposed outside the middleware's encrypted credential store.

Endpoint and headers. POST to https://a.klaviyo.com/api/events. Required headers: Authorization: Klaviyo-API-Key pk_xxx, revision: 2026-04-15, Content-Type: application/json, Accept: application/vnd.api+json. Rate limits: 350/s burst, 3500/m steady.
  • Make.comAdd the key to Scenarios → Connections. The built-in Klaviyo app is preferred. Make stores credentials in its encrypted Connections vault and references them by connection ID, never inlining the key into scenario JSON. Make is migrating Klaviyo to OAuth2, so use OAuth where it's offered.
  • Make HTTPIf using the raw HTTP module, create a Keychain entry of type "API Key Auth" and reference it from the module's Authorization field. Never type pk_xxx into the Headers tab. That value gets serialised into blueprint exports.
  • n8nCreate a Header Auth credential: Name = Authorization, Value = Klaviyo-API-Key pk_xxx. Attach it to the HTTP Request node via the Credential dropdown. Don't paste the key into the node's Headers parameter directly. Credentials are AES-encrypted at rest using N8N_ENCRYPTION_KEY and excluded from workflow exports by default.
  • PipedriveThe key never enters Pipedrive. Not as a custom field, not as a webhook URL parameter, not in deal notes, not in Automations action bodies. Pipedrive fires the webhook; the middleware injects the credential server-side.
  • NeverNo query-string secrets (?api_key=), no hardcoded Authorization values in node JSON, no committing scenario blueprints or workflow exports that contain credential payloads, no full-header logging in execution retention, no screenshots of the unmasked Connections panel.
  • RotationKlaviyo keys cannot be edited, so rotation means creating a new key with identical scopes, updating the Make Connection or n8n Credential to point at it, then deleting the old key. If the key is ever exposed (committed, pasted, leaked), revoke immediately in Klaviyo and let us know so we can issue a fresh one.
